AIエージェント用 要件定義書 v1.0（経営者サークルLP）
0. このドキュメントの目的

本ドキュメントは、AIエージェントが追加質問を最小限にしつつ、ズレなく設計・実装できるようにするための要件定義である。
不明点がある場合、推測せず質問する。

1. プロジェクト概要
1-1. 種別

経営者サークルのLP（メディア型）＋会員紹介（SEO前提）

1-2. 目的（Why）

サークルの世界観・価値を伝える

会員のストーリー（8割）を通じて共感と信頼を生む

説明 → 共感 → 申込み の流れで自然に入会/問い合わせへ誘導する

1-3. サークル基本情報

【名称】
ガイアの酒場

【ミッション】
私たちは、志ある経営者同士が
能力と人脈を持ち寄り、
課題解決と価値創造を共に行う場をつくります。

ここは、売り込みの場ではありません。
同じ未来を目指す仲間が集い、
次の挑戦へ向かうための"酒場"です。

【ベネフィット】紹介制 × 共創 × 事業提携
入会で得られること：
- 紹介制だからこそ生まれる、質の高い人脈
- 売り込みではなく、課題相談から始まる関係
- 自然な流れで生まれる、事業提携と共創

※補足：GIA は、名刺交換の場ではありません。誰かの課題に、誰かの得意が自然と集まる場です。

1-4. ゴール（KGI）

入会申込み（フォーム/決済まで）

問い合わせ（LINE/フォーム等）
※両方を最終ゴールとする（申込み優先だが問い合わせも残す）

2. 想定ユーザー

入会検討中の経営者・個人事業主

条件より「人」「思想」「場の空気感」で判断する層

スマホ閲覧が主（モバイルファースト）

3. 成功の定義（最重要）

初見で「どんなサークルか」が短時間で理解できる

会員一覧の多様性が伝わり「この人たちと関わりたい」が生まれる

会員詳細ページがURL共有可能でSEO前提（個別ページ）

会員詳細からサークルへの入会導線が自然につながる

「売り込み臭」「AI感」を出さない（読み物として成立）

4. 情報設計（画面構成）
4-1. 公開ページ

LPトップ

Hero（世界観・コピー）

サークル概要（価値・対象・得られること）

会員一覧（カード一覧＋検索）

入会導線（説明→共感→申込み）

問い合わせ導線（サブ）

会員一覧（LP内セクション or /members でも可）

会員カードの並び

検索（名前/カテゴリ/仕事）

タグ（複数軸）

会員詳細ページ（/members/[slug]）

ストーリー（8割）

サービス（2割）

条件付き：外部導線（個別問い合わせ等）

サークル入会導線（必須）

入会ページ（/join）

入会の説明、規約/注意

申込みフォーム

決済（方式は後述の「未確定事項」にて質問）

問い合わせページ（/contact）

フォーム or LINE誘導（方式は後述の「未確定事項」にて質問）

5. 機能要件
5-1. MUST（必須）

LPトップ（説明→共感→申込み）

会員一覧表示（初期10〜30、将来100+）

会員詳細ページ（SEO/URL共有）

検索：名前 / カテゴリ / 仕事（職種・提供サービス・肩書）

入会導線（会員詳細にも設置）

問い合わせ導線

運営用 管理画面（最低限）

会員CRUD（作成/編集/削除）

公開ステータス管理（下書き/公開/非公開）

タグ編集

外部導線表示ON/OFF（個別問い合わせリンク等）

5-2. WANT（あれば嬉しい）

タグによるフィルタ（複数選択）

並び替え（新着/おすすめ 等）

OG画像/メタ情報の最適化（SNS共有）

5-3. DO NOT（今回はやらない）

会員本人がログインして編集する仕組み（現時点では不要）

AIが前面に出るUI/文言

スペック比較表・数字訴求中心のLP

6. UX / UI 方針（デザイン判断ルール）

「売り込み感」を消す：余白、読みやすい本文、落ち着いたトーン
スタイリッシュに最高のデザインで。
超一流のWEBデザイナーとして適切にデザインして。

ストーリーは“読み物”として成立させる

CTAは強すぎない表現（例：「参加する」「話を聞いてみる」）

スマホ最適化前提（長文でも読める）

7. コンテンツ要件（ストーリー設計）
7-1. ストーリー型（確定）

起点 → 転機 → 今 → 未来（王道）

7-2. ヒアリング質問テンプレ（運営が使用）

起点：背景/始めた理由/当時の悩み

転機：価値観が変わった出来事/決定的な変化

今：今大事にしている姿勢・信条

未来：作りたい世界/サークルで実現したいこと

補足：印象的なエピソード、よく言われる特徴

※運営がヒアリング→文章化→公開する。

8. データ設計（Supabase想定・概念）
8-1. Members（会員）

id

slug（URL用、ユニーク）

name

photo_url（必須）

headline（短い一言）

role_title / job_title

story_origin（起点）

story_turning_point（転機）

story_now（今）

story_future（未来）

services_summary（サービス2割）

tags（多対多推奨：member_tags）

status（draft / published / private）

allow_direct_contact（bool）

contact_links（json：line / site / sns 等、表示許可があるもののみ）

created_at / updated_at

8-2. Tags（タグ）

id

name

type（industry / region / challenge / strength など）

sort_order（任意）

8-3. MemberTags（中間）

member_id

tag_id

9. 運用要件（ワークフロー）

運営がヒアリング・編集し、公開品質を担保する

公開ステータスで制御（draft→published、またはprivate）

直接問い合わせ導線は条件付き：運営がON/OFF管理
（デフォルトOFF）

10. 技術スタック（確定）

フロント：Next.js（React）

DB/Auth/Storage：Supabase

画像：Supabase Storage（想定）

デプロイ：Vercel（想定）

11. 未確定事項（AIが質問すべき事項）

AIエージェントは以下を質問して確定させること：

入会の決済方式（Stripe等）を使うか、まずはフォームのみか

問い合わせ導線の実体（LINE固定？フォーム？両方？）

デザインテイスト（例：高級/親しみ/ポップなど）とブランドカラー

会員詳細の外部リンク種類（LINE/サイト/SNSのどれを許可するか）

会員の公開範囲（誰でも閲覧可/限定公開の必要有無）

AIエージェントへの行動指針

不明点は推測しない。質問する。

UX（読みやすさ・共感）を優先し、機能追加で複雑化しない。

SEO前提：members/[slug] のメタ/OG最適化を考慮。

初期10〜30人で快適、将来100+でも破綻しない構造（タグ/検索）。
